# Требования Gatherly

## Функциональные требования

- Посетитель может зарегистрироваться, указав username, email, password, bio и avatar URL.
- Зарегистрированный пользователь может войти и получить простой bearer token.
- Пользователь может просматривать профили и список пользователей.
- Пользователь может подписаться на другого пользователя и отписаться от него.
- Пользователь не может подписаться на самого себя или создать повторную подписку.
- Организатор может создать событие в статусе draft.
- Только организатор может обновить, опубликовать или отменить событие.
- В публичном списке отображаются только опубликованные публичные события.
- Feed содержит опубликованные события пользователей, на которых подписан текущий пользователь.
- Private-события видны только организатору и участникам.
- Followers-only события видны подписчикам и участникам.
- Пользователи могут присоединяться к событию, выходить из него и менять статус участия.
- Повторные записи участия запрещены.
- Если событие заполнено, новые участники попадают в waitlist.
- Если участник со статусом GOING выходит, первый пользователь из waitlist повышается до GOING.
- Подписчики получают уведомления, когда пользователь, на которого они подписаны, публикует событие.
- Участники получают уведомления при изменении времени или места события.
- Участники получают уведомления при отмене события.
- Доставка уведомлений использует outbox-like таблицу и поддерживает режимы отказа/восстановления.

## Нефункциональные требования

- MVP должен запускаться локально через CMake и современный C++ compiler.
- API должен возвращать понятные HTTP status codes и JSON-ошибки.
- Бизнес-логика должна тестироваться без запуска HTTP-сервера.
- Отказ доставки уведомлений не должен откатывать публикацию события.
- Репозиторий должен быть готов к будущей интеграции PostgreSQL.
- Система должна быть понятной для обсуждения на backend и system design собеседованиях.
