# Требования Gatherly

## Функциональные требования

- Пользователь может зарегистрироваться и войти в систему.
- Пользователь может просматривать профили и список пользователей.
- Пользователь может подписываться на других пользователей и отписываться от них.
- Пользователь не может подписаться на самого себя или создать повторную подписку.
- Организатор может создать событие, изменить его, опубликовать или отменить.
- В общем списке показываются только опубликованные публичные события.
- Пользователь видит отдельную ленту событий от людей, на которых он подписан.
- Пользователь может открывать события, записываться на них, выходить из них и менять свой статус участия.
- Если мест в событии больше нет, новые участники попадают в лист ожидания.
- Если кто-то выходит из события, первый человек из листа ожидания получает место.
- Пользователи получают уведомления о публикации, обновлении и отмене событий, которые для них важны.

## Нефункциональные требования

- Проект должен локально собираться и запускать unit-тесты через CMake.
- Основной запуск приложения должен работать через Docker Compose.
- Данные приложения должны храниться в PostgreSQL.
- API должен возвращать понятные HTTP status codes и JSON-ошибки.
- Бизнес-логика должна проверяться тестами отдельно от HTTP-слоя.
- Ошибка доставки уведомления не должна ломать основное действие пользователя.
- Документация должна оставаться короткой и понятной.
